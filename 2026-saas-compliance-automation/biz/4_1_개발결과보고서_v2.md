## v1 한계 및 v2 개선 매핑

> 과업지시서 v2 변경 이력(v1→v2)과 정합. v1에서 부족했던 항목 ↔ v2 해결 방식 1:1.

| # | v1 한계 (부족했던 점) | v2 개선 (해결 방식) | 입증 캡처 |
|:---:|:---|:---|:---|
| 1 | **단일 프레임워크** — ISMS-P 통제만, CSAP·ISO27001 미지원 | 공통 통제 16개 모델 + 3개 프레임워크 매핑표, 단일 입력 시 매핑된 전 프레임워크에 동시 반영, 프레임워크 전환 시 공통 통제 상태 재사용 | 5.1, 5.2 |
| 2 | **단일 사용자·단일 조직** — 데이터 격리 개념 없음 | 멀티테넌트 2개 시드 + 신규 조직 추가, 조직별 통제·증적·티켓·정책·감사로그 완전 격리(타 조직 0% vs 53.1%) | 5.11, 5.14 |
| 3 | **증적 수동 업로드만** — 자동 수집 없음 | AWS·Azure·GCP 클라우드 설정 스캔 mock → 통제 자동 매핑 → 증적 후보 자동 등록 → 통제 상태 전파 | 5.4, 5.5 |
| 4 | **갭분석은 표시만** — 조치 워크플로 부재 | 위험가중 알고리즘으로 갭 우선순위화 → 티켓 생성·담당자 배정·칸반 상태 전이 → 완료 시 통제 충족 전파 | 5.6 |
| 5 | **산출물이 정책 .md 다운로드 한정** — 심사 제출물 부재 | 통제·증적·정책·준비율 종합 심사 대응 패키지 PDF 실 생성(한글 깨짐 0) + CSV·알림 발송 | 5.9 |
| 6 | **정책 단일 버전** — 이력·비교·롤백 없음 | 정책 버전 누적 + 라인 diff(추가/삭제 색상) + 버전 조회 + 이전 버전 롤백 | 5.7, 5.8 |
| 7 | **무권한 단일 뷰** — 역할 분기 없음 | CISO·실무자·심사관 3역할, 역할별 메뉴·액션 노출 분기, 심사관 읽기 전용(상태 변경 비활성) | 5.12 |
| 8 | **준비율 스냅샷 1점** — 추세·감사 추적 부재 | 준비율 시계열 추세 차트 + 전 액션 감사로그(역할·시각·내용) 기록·조회·CSV | 5.10 |

## v2 신규/심화 산출물

> 8가지 핵심 산출물을 캡처로 입증한다. 각 캡처는 실제 `file://` 구동 화면이다.

### #1 다중 프레임워크 매핑 — 단일 입력 → 다중 동시 반영

![공통 통제 16개의 ISMS-P·CSAP·ISO27001 매핑표와 단일 입력 데모, U-GOV-01 충족 반영](./captures/v2/v2_01_framework_mapping.png)

무엇을 보여주는가: 공통 통제 16개가 ISMS-P·CSAP·ISO27001 control_id로 매핑된 표와, "단일 입력 → 다중 프레임워크 동시 반영 데모"에서 U-GOV-01을 충족 처리하자 우측 토스트 "ISMS-P, CSAP, ISO 27001 동시 반영" 및 현재상태 "충족" 배지 표기.
의도: 공통 통제 1건 입력이 매핑된 모든 프레임워크에 동시 반영됨(§5.1 #1)을 입증.
검토 결과: 매핑표 control_id·도메인·현재상태 정상, 동시 반영 토스트·배지 정상, 한글 정상.

### #1 (심화) 프레임워크 전환 — 공통 통제 상태 재사용

![ISO 27001 활성 전환 후에도 U-GOV-01 충족 상태가 유지된 매핑표](./captures/v2/v2_02_framework_switch_iso.png)

무엇을 보여주는가: ISO 27001로 활성 프레임워크를 전환(상단 카드·좌하단 "프레임워크: ISO 27001"·토스트 "ISO 27001 활성화 (공통 통제 재사용)")해도 U-GOV-01이 "충족"을 유지.
의도: 1회 입력·다중 재사용(제안서 §3.2 차별점·§5.3 전환비용 해자)을 동작으로 입증.
검토 결과: 활성 프레임워크 표기·공통통제 상태 재사용 정상, 한글 정상.

### #2 다중 조직(테넌트) — 데이터 격리

![클라우드원(CSAP 대상) 테넌트로 전환 후 14개 CSAP 통제·0% 준비율로 격리된 대시보드](./captures/v2/v2_11_tenant_switch_csap.png)

의도: 2개 이상 조직 생성·전환·데이터 격리(§5.1 #2)를 입증.

### #3 증적 자동수집 커넥터 mock — 클라우드 설정 스캔

의도: 클라우드 설정 스캔 mock → 증적 후보 자동 등록 → 통제 매핑·스캔 로그(§5.1 #3, 외부 시스템 통합 #1)를 입증.
검토 결과: 프로바이더별 스캔 결과·통과/미흡·매핑 통제·raw값 정상, 한글 정상.

| 역할 권한 | CISO·실무자·심사관 3역할, 메뉴·액션 권한 분기, 심사관 읽기 전용 | 실 동작 |
| 준비율 추세·감사로그 | 준비율 시계열 SVG 추세, 전 액션 감사로그·CSV | 실 동작 |

- 뷰 9종(대시보드·다중프레임워크·체크리스트·자동수집스캔·증적·티켓·정책·심사패키지·감사로그) — 모두 인터랙션 가능(8종+ 충족).
- 워크플로 3개+: (a) 프레임워크→통제→증적→정책→패키지 준비 흐름, (b) 갭→티켓 생성→담당자 배정→진행→완료→충족 전파, (c) 정책 생성→새 버전→diff→롤백.
- 실 알고리즘: 가중 준비율 산식 + 위험가중 갭 우선순위(상태×도메인위험×증적부재) + 정책 라인 diff.
- 외부 시스템 통합 2건+: 클라우드 스캔 커넥터 mock(설정값 수집), CSV 입출력(통제 현황·감사로그), 알림 발송 mock 로그, PDF 파일 출력.
- 토스트는 mock 대체가 아닌 실 액션 후 피드백이며, 모든 액션은 상태 변경·영속·재렌더로 실 반영.
