## 2. 구현/제작 범위

- **뷰 5종**: ① 재고 대시보드 ② 품목·공급사 관리 ③ 소비예측·재주문 제안 ④ 발주서·발송 이력 ⑤ 입고·정산.
- **워크플로 5단계**: 재고 입력/소비 기록 → 소비예측(이동평균·σ) → 재주문 제안(안전재고·ROP·부족분) → 발주서 생성·발송(mock) → 입고확정·정산.
- **실 알고리즘**: 이동평균 μ, 표본표준편차 σ, 안전재고 = Z·σ·√L, ROP = μL + 안전재고, 목표재고 = μ·커버일수 + 안전재고, 제안수량 = (목표−현재고)→MOQ 올림.
- **실 동작(목업 아님)**: 품목 CRUD·소비 기록(재고 차감)은 localStorage에 즉시 반영, 차트(Chart.js)는 실 데이터로 렌더, 발주서는 jsPDF로 실제 PDF 생성·다운로드, 발송은 mock 알림톡 로그, 입고는 재고 증가+정산 누적.
- **반응형**: 데스크톱 사이드바 248px, 모바일 5탭 바텀바 + 햄버거 오프캔버스 드로어, KPI 2열 스택, 차트 고정높이.

## 4. 실행/구동 방법

```bash
# 1) 앱 직접 열기 (빌드 불필요)
open projects/food-supply/v1.html      # 또는 브라우저로 file:// 로 열기

# 2) 캡처 재현 (Playwright 임시 설치 후 삭제)
cd projects/food-supply
npm i -D playwright && npx playwright install chromium
node capture.mjs                       # captures/v1, captures/mobile/v1 생성
rm -rf node_modules package*.json      # 잔존물 제거
```

> 외부 네트워크 없이도 핵심 기능 동작(차트·예측·localStorage). 발주서 한글 폰트는 CDN fetch, 실패 시 canvas 래스터로 폴백해 깨짐을 방지한다.

### 5.1 PC (1280×800) — `./captures/v1/`

**① 재고 대시보드**

**② 품목·공급사 관리**
![PC 품목관리](./captures/v1/02-catalog.png)
무엇: 품목 추가/수정 폼(현재고·단가·리드타임·MOQ·Z), 공급사 표, 품목 목록. / 의도: 발주 파라미터를 직접 관리. / 검토: 폼·표가 2열 그리드로 정렬, 한글 정상.

**③ 소비 기록(재고 차감)**
![PC 소비기록](./captures/v1/03-catalog-consume.png)
무엇: [−소비] 클릭으로 재고가 차감되고 소비 이력에 기록됨(토스트). / 의도: 실 데이터 변경이 localStorage에 반영. / 검토: 토스트 "n단위 소비 기록 (재고 …)" 노출, 목업 아님.

**④ 소비예측·재주문 계산**
![PC 예측](./captures/v1/04-forecast.png)
무엇: [예측 실행] 후 품목별 μ·σ·안전재고·ROP·현재고·목표재고·제안수량·판정 표. / 의도: 알고리즘이 실제로 계산됨을 입증. / 검토: 예) 양파 μ13.00·σ5.07·안전재고11.8·ROP37.8·제안75, "발주 필요"/"유지" 판정이 ROP 비교로 정확.

**⑤ 발주 제안 카드**
![PC 제안](./captures/v1/05-suggest.png)
무엇: 재주문점 이하 품목별 제안 카드(근거: 현재고≤ROP·일평균·안전재고·목표), 수량 조정 시 금액 실시간 갱신. / 의도: "왜 이만큼 발주하는가"를 근거와 함께 제시. / 검토: 수량 입력에 따라 라인 합계 즉시 재계산.

**⑥ 발주서 초안(공급사별)**
![PC 발주서초안](./captures/v1/06-draft.png)
무엇: 공급사별로 묶인 발주서 초안(품목·수량·단가·금액·합계) + PDF 다운로드 + 발송(mock) 버튼. / 의도: 제안이 정식 발주서로 구조화됨. / 검토: 한밭청과·대구축산유통·신선수산으로 그룹핑, 합계 정확.

**⑦ 발송 이력 + mock 알림톡 로그**
![PC 발송이력](./captures/v1/07-sent-log.png)
무엇: 발주번호(PO-YYYYMMDD-seq)·공급사·금액·상태·일시 이력 표 + mock 발송 로그(채널 카카오 알림톡, 수신처, HTTP 200 mock). / 의도: 발송이 추적 가능한 이벤트로 기록됨. / 검토: 3건 발송, 로그에 공급사 연락처·발주번호·결과 표시.

**⑧ 입고 대기**
![PC 입고대기](./captures/v1/08-settle-pending.png)
무엇: 발송했으나 미입고인 발주의 입고확정 대기 목록. / 의도: 입고 전/후 상태 분리. / 검토: KPI(발송 대기/입고 완료/누적 정산액)와 대기 표가 일치.

**⑨ 입고확정·정산**
![PC 정산](./captures/v1/09-settled.png)
무엇: [입고확정] 후 재고 증가 + 정산 누적, 누적 정산액 ₩3,286,800, 정산 내역 표(발주번호·공급사·정산액·입고일시). / 의도: 워크플로 종착점(정산)까지 동작. / 검토: 입고 완료 3건, 정산 내역 시각 기록.

**⑩ 상태 지속성(새로고침 후)**
![PC 지속성](./captures/v1/10-persist.png)
무엇: 새로고침 후 대시보드 — 재주문 필요 0, 정산 ₩3,196,800, 재고 증가(양파 104) 유지. / 의도: localStorage 지속 입증. / 검토: 앞선 발주·입고·정산 결과가 새 세션에서도 보존됨.
