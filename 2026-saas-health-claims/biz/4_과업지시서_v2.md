# 과업지시서 v2 — 클레임브릿지 청구 자동화 플랫폼 심화(베타)

> v1([`3_과업지시서_v1.md`](./3_과업지시서_v1.md)) 검수 후 추가/변경된 과업. v1을 수정하지 않고 신규 파일로 분리. 본 문서는 [`2_개발계획서.md`](./2_개발계획서.md) §작업분해의 v2 외주 구분 행을 §1 범위로 옮기고, v2 자체 심화 산출물을 §5에 명시한다.

## 1. 과업의 개요

### 1.1 과업의 목적

1) **외주에 맡기는 직접 목적**: v2 베타에서 다중 기관(테넌트)·다중 사용자·API mock·전자서명을 지탱하는 **멀티테넌트 백엔드 데이터 스키마 설계 자문 + 청구서 PDF 엔진 고도화**(다중 기관 양식·실손 청구서 추가 레이아웃)를 외주 위탁한다.
2) **사업·제안 연결**: [`1_제안서.md`](./1_제안서.md) §8 로드맵 Beta(v2)·§10.1 멀티테넌트 설계, §11.3 마일스톤 M4~6에 해당. 단일 처방 데이터 위에 다중 기관·역할 권한을 얹는 것이 §5.3 해자 2·3층의 기술 기반이다.

### 1.2 과업의 범위

**외주사가 수행할 업무 (In Scope)**

1) 멀티테넌트 데이터 스키마 설계 자문(shared-schema + Row-Level Security 모델, `tenant_id` 격리 설계서).
2) 역할 권한(의료진·청구담당·환자) RBAC 권한 모델 설계 자문.
3) 청구서 PDF 엔진 고도화 — 다중 기관 양식 분기·실손 청구서 레이아웃 추가.
4) 청구 상태머신(제출→접수→지급/반려) 데이터 모델 설계 자문.
5) 설계서·ER 다이어그램·인계 문서.

**범위 외 (Out of Scope)**

- 8대 v2 자체 산출물의 프론트 구현 (자체 개발 — §5 참조).
- 코드 자동매핑 알고리즘 로직 (자체 개발).
- 실 심평원/보험사 API 연계 (v3 별도 과업, mock은 자체).
- ISMS-P 인증 취득 실행 (v3).

### 1.3 자체 개발 v2 산출물 개요

§5 성과품 목록의 8대 자체 산출물이 v2 핵심이며, 외주는 그 기반 스키마·PDF 엣지케이스를 보강한다.

## 2. 과업의 세부 내용

### 2.1 멀티테넌트 스키마 설계 사양

| 항목 | 사양 |
|:---|:---|
| 격리 모델 | shared-schema + RLS, 전 테이블 `tenant_id` 강제 |
| 핵심 엔티티 | Tenant·User(역할)·Patient·Prescription·Claim·ClaimEvent(상태이벤트)·Consent |
| 민감정보 | 환자 식별·주민번호급 컬럼 암호화 설계 명시 |
| 상태머신 | Claim 상태(작성→제출→접수→지급/반려) 전이 규칙 정의 |

### 2.2 PDF 엔진 고도화 사양

- 기관별 청구서 양식 분기(로고·기관정보 변수화).
- 실손 청구서 레이아웃(보험사 제출용) 추가.
- v1 청구서 엔진과 하위호환 유지.

### 2.3 품질·성능 기준 (정량)

| 기준 | 목표 |
|:---|:---|
| 테넌트 격리 설계 | 타 테넌트 데이터 조회 0건(설계 검증) |

### 5.2 자체 v2 핵심 산출물 8가지 (발주자 자체 개발 — 검수 기준)

| # | 산출물 | 검수 합격 조건 (측정 가능) |
|:--:|:---|:---|
| 1 | **반응형(데스크톱 + 모바일 환자뷰)** | 데스크톱·모바일 뷰포트 각각 정상 렌더, 환자 전용 모바일 뷰 동작 |
| 2 | **다중 기관(테넌트)** | 기관 2곳 이상 데이터 분리, 기관 전환 시 데이터 격리 확인 |
| 3 | **청구코드 자동매핑 알고리즘(처방→EDI 코드)·검증** | 처방 입력 → 수가/급여/EDI 코드 자동 매핑, 매핑 근거 표기, 검증 룰 통과/실패 표시 |
| 4 | **심평원/보험사 청구 API mock(제출→접수→지급)** | 청구 제출 → mock 응답으로 접수→지급 상태 전이, 응답 로그 표시 |
| 5 | **환자 onboarding 동의·전자서명** | 동의 화면 → 캔버스 서명 → 이미지 저장, 동의 이력 기록 |
| 6 | **실손 청구 자동화 흐름 mock** | 실손 대상 청구 → 보험사 mock 제출 → 처리 상태 추적 |
| 7 | **역할 권한(의료진·청구담당·환자)** | 역할별 로그인 시 화면·기능 분기(권한 없는 액션 차단) |
| 8 | **청구 상태·지급률·반려사유 대시보드** | 청구 상태별 집계·지급률(%)·반려사유 목록 시각화 |

> 자체 산출물은 화면/뷰 8종 이상·워크플로 3개 이상·신규 캡처 10장 이상을 [`4_1_개발결과보고서_v2.md`](./4_1_개발결과보고서_v2.md) 에서 실 구동 캡처로 입증한다(CLAUDE.md §2.4 v2 기준).

## 6. 변경 이력 (v1 → v2)

| 구분 | v1 | v2 변경/추가 |
|:---|:---|:---|
| 청구 워크플로 | 단일 기관·단일 사용자 처방→청구→상태추적 | 다중 기관(테넌트)·역할 권한(의료진·청구담당·환자) 추가 |
