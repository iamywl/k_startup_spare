# 과업지시서 v1 — 클레임브릿지 처방→청구 자동화 시제품(PoC)

> 본 문서는 [`2_개발계획서.md`](./2_개발계획서.md) §작업분해의 v1 외주 구분 행을 §1 과업의 범위로 옮겨 작성한다. 자체 개발 v1 산출물도 명시한다.

## 1. 과업의 개요

### 1.1 과업의 목적

1) **외주에 맡기는 직접 목적**: 클레임브릿지 v1 시제품(웹앱)의 **한글 청구서 PDF 출력 엔진**(한글 폰트 임베드·청구서 레이아웃·QR 검증 코드) 제작. 자체 개발한 처방→청구 워크플로 SPA가 생성한 청구 데이터를 인쇄·보관 가능한 PDF로 출력하는 모듈을 외주 위탁한다.
2) **사업·제안 연결**: 본 과업은 [`1_제안서.md`](./1_제안서.md) §2.2 "청구서 생성·제출" 모듈, §8 로드맵 MVP(v1) 단계의 구성요소다. 청구서 PDF 출력은 의료기관 청구 실무의 필수 산출물이며, v1 PoC가 "동작하는 청구 워크플로"임을 입증하는 핵심 증거다.

### 1.2 과업의 범위

**외주사가 수행할 업무 (In Scope)**

1) 청구서 PDF 생성 엔진 구현(jsPDF 기반, 한글 폰트 임베드).
2) 청구서 표준 레이아웃 디자인·구현(환자 정보·처방·청구코드·수가·합계).
3) 청구 검증용 QR 코드 생성·삽입.
4) 자체 SPA가 전달하는 청구 데이터(JSON) → PDF 변환 API 모듈화 및 인계.
5) 단위 테스트 + 사용 가이드 문서.

**범위 외 (Out of Scope) — 외주사가 수행하지 않음**

- 처방→환자등록→코드매핑→청구서→상태추적 워크플로 로직 (자체 개발).
- 환자 목록·청구 대시보드 화면 (자체 개발).
- localStorage 상태 지속 (자체 개발).
- 청구코드 자동매핑 알고리즘 (v2 자체).
- 실 심평원/보험사 API 연계 (v3, 별도 과업).
- 서버·DB·인증 인프라 (v1 미해당).

### 1.3 자체 개발 v1 산출물 (외주 범위 외, 발주자 자체 수행)

| 산출물 | 내용 |
|:---|:---|
| 다단계 워크플로 | 처방 입력 → 환자 등록 → 청구코드(수가/급여) 매핑 → 청구서 생성 → 상태 추적 (4단계+) |
| 환자 목록 화면 | 환자 CRUD·검색 |
| 청구 대시보드 | 청구 상태별 집계·필터 |
| 상태 지속 | localStorage 기반 새로고침 후 유지 |

## 2. 과업의 세부 내용

### 2.1 청구서 PDF 출력 사양

| 항목 | 사양 |
|:---|:---|
| 라이브러리 | jsPDF (또는 동등) + 한글 폰트(예: Noto Sans KR) 임베드 |
| 출력 항목 | 의료기관명·환자 식별·처방 내역·청구코드(수가/급여 구분)·금액·합계·청구일자 |
| 한글 | 폰트 임베드로 한글 깨짐 0건 (검수 필수) |
| QR | 청구 식별값을 담은 QR 코드 청구서 우상단 삽입 |
| 입력 인터페이스 | 자체 SPA가 전달하는 청구 데이터 JSON 스키마 수신(스키마는 발주자 제공) |
| 출력 | 브라우저에서 PDF 다운로드 + 미리보기 |

### 2.2 UI/UX 제작 사양

- PDF 미리보기 버튼·다운로드 버튼은 자체 SPA의 "청구서 생성" 단계에 임베드 가능한 컴포넌트/함수 형태.
- 디자인 가이드: 의료 청구서 톤(흑백 인쇄 가독성 우선, 표 정렬 명확).

### 2.3 사용 부품·의존성

- jsPDF, 한글 웹폰트, QR 생성 라이브러리(예: qrcode). 검증 안 된 라이브러리 무분별 추가 금지(CLAUDE.md §3.2).

### 2.4 품질·성능 기준 (정량)

| 기준 | 목표 |
|:---|:---|
| 한글 렌더링 | 깨짐 0건 |
| PDF 생성 시간 | 청구서 1건 ≤ 2초 |
| QR 스캔 | 표준 QR 리더로 식별값 100% 디코딩 |
| JSON 스키마 호환 | 발주자 제공 스키마 100% 처리 |
