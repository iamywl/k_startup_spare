## 1. 과업의 개요

### 과업의 목적

1) **직접 목적** — 중소 장기요양기관(요양원·주야간보호·방문요양) 운영 SaaS 「돌봄노트」의 **시제품(PoC) 1식**을 제작한다. 시설장·사회복지사·요양보호사·보호자 4역할이 사용하는 단일 웹 애플리케이션으로, 현장 라운드 1회 입력이 돌봄기록·급여청구·평가근거·보호자 공유로 동시에 분기하는 핵심 흐름을 실제 동작으로 구현한다.
2) **사업 연결** — 본 시제품은 [`1_제안서.md`](./1_제안서.md) §2 Solution(S-2 핵심 기능·S-3 핵심 워크플로·S-4 PoC 아키텍처)와 §차별화 기술의 구매동인 논증 ⑤(데모 정합)에서 약속한 "1회 입력 → 청구·평가·보호자 동시 충족"을 실증하는 산출물이다. 심사·시연 환경에서 인터넷·서버 없이 단독 구동되어야 한다.

### 과업의 범위

1) **수행 업무(In Scope)**
   - 4역할 RBAC·뷰 라우팅·한/영 i18n·localStorage 상태 지속(시드·마이그레이션 포함) 골격 구축
   - 운영 대시보드: KPI 4종, 등급 분포 도넛, 오늘의 방문 목록, 위험 자동 스코어링 테이블
   - 어르신 관리: 카드 그리드·실시간 검색·등급 필터·상세 모달·신규 등록·CSV·요약 PDF
   - 케어플랜 5단계 워크플로: 대상 선택·기간·목표 CRUD·급여 서비스 추가(코드·횟수→금액 자동계산)·발행+PDF
   - 방문기록 4단계 워크플로: 어르신/서비스/시간 선택·서비스 체크리스트(7항목)·현장 사진 실제 업로드·특이사항·상태 선택·저장 시 보호자 알림 자동생성
   - 급여청구·정산: 코드×수가×횟수 자동 산정, 등급별 월 한도·초과 전액본인부담·본인부담률·공단부담 계산, 5단계 청구 워크플로(Mock 공단 전송·접수번호 발급)·수납 처리·명세서 PDF
   - 스케줄·배정, 근태(출퇴근·근무시간 자동계산·CSV), 알림센터, 리포트·통계(막대·도넛 차트·CSV), 외부연동(Mock 로그), 기관설정(편집·JSON 백업·초기화)
   - 보호자 역할 4뷰: 가족 대시보드·돌봄일지·청구납부
   - jsPDF 한글 문서 4종(케어플랜·청구서·방문일지·어르신요약)
   - 모바일/PC 반응형: 데스크톱 사이드바 ↔ 모바일 바텀 탭바 + 오프캔버스 드로어
   - 실 구동 캡처(PC·모바일 분리) 및 검수 협조
2) **Out of Scope(범위 외)** — 분쟁 예방을 위해 명시한다.
   - 실 백엔드 서버·DB·다기관(멀티테넌트) 인프라 구축 (v2 이후)
   - 공단 전자청구 실 연동·실 알림톡 발송·실 OAuth/SSO (PoC는 Mock 로그로 대체)
   - 회원가입·로그인 정식 인증(시제품은 역할 전환으로 우회, `CLAUDE.md` §3.4)
   - 모바일 네이티브 앱(iOS/Android) 패키징
   - 보안 인증(ISMS 등)·개인정보 실 운영 처리·법적 자문

## 2. 과업의 세부 내용

### 2.1 시제품 사양
- **구동 형태**: 단일 HTML 파일(`v1.html`), 빌드리스·오프라인 단독 구동. `<meta name="viewport" content="width=device-width, initial-scale=1">` 필수.
- **상태 저장**: localStorage 기반. 최초 진입 시 데모 데이터 시드, 새로고침 후에도 입력 유지. 스키마 변경 시 자동 마이그레이션.
- **역할/권한**: 관리자·사회복지사·요양보호사·보호자 4역할. 역할별 접근 뷰가 분기(RBAC)되어야 한다.
- **급여 산정 정확성**: 급여비 = 서비스코드 단가 × 횟수, 등급별 월 한도 초과분 전액 본인부담, 유형별 본인부담률 적용, 공단부담액 = 총액 − 본인부담. 합계가 명세·정산·대시보드 KPI 간 일치해야 한다.

### 2.2 사용자 UI/UX 제작 사양
- 디자인 톤: [`design-guide/디자인_애플_토스.md`](./design-guide/디자인_애플_토스.md)(Apple×Toss) 준수. 모바일=[`디자인_모바일.md`], 데스크톱=[`디자인_PC.md`].
- 화면 흐름: 다단계 워크플로(케어플랜 5단계, 방문기록 4단계, 청구 5단계)는 진행률 표시·이전/다음 내비게이션을 제공.
- 반응형: 좁은 화면(360~430px) 가로 overflow 0, 데스크톱 사이드바는 모바일에서 바텀 탭바 + 더보기 오프캔버스 드로어로 재구성(축소판 금지). 출고 전 [`반응형_검수_필수.md`] 통과.

### 2.3 사용 부품·라이브러리·외부 의존성
- jsPDF 2.5.1 (CDN). 부재 시 mock 안내로 폴백(오프라인 시연 보장). 한글은 캔버스 렌더 후 이미지 삽입으로 깨짐 방지.
- 차트는 외부 차트 라이브러리 없이 자체 SVG/CSS 구현.
- 공단 청구 API·카카오 알림톡은 Mock(키 없이 로그). API 키 하드코딩 금지(`CLAUDE.md` §3.1).

### 2.4 품질·성능 기준 (정량)
- 구현 기능 수: **120개 이상** (코드 하단 `FEATURES[]` 배열·`window.__FEATURES.length`로 검증 가능).
- 화면/뷰: **17종 이상**. 다단계 워크플로 **3개 이상**.
- 상태 지속성: 새로고침 후 입력 데이터 100% 유지.
- 한글 표시: 화면·PDF 산출물 한글 깨짐 0건.
- 반응형: 390px·1280px 뷰포트에서 가로 overflow 0, 레이아웃 깨짐(사이드바 하이재킹·잘림) 0건.
- 콘솔 에러(pageerror) 0건.
