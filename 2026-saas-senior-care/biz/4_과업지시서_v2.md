## 1. 과업의 개요

### 과업의 목적

1) **직접 목적** — v1 PoC(1회 입력 → 청구·평가·보호자 동시 분기)를 **베타 출시·시리즈A 데모 가능 수준**으로 심화한다. 구체적으로 ① 다기관(멀티테넌트) 법인 본부 콘솔, ② 실 알고리즘 2종(낙상 위험 **로지스틱 회귀** 예측 모델 / 인력배치 **최적화 휴리스틱**), ③ 외부 시스템 통합 2건 이상(국민건강보험공단 **EDI 전자청구 + OAuth** Mock / **CSV 외부 입출력**), ④ 감사 로그·수가 시뮬레이션을 v1 단일 HTML 위에 추가 구현한다.
2) **사업 연결** — 본 심화 산출물은 [`1_제안서.md`](./1_제안서.md) §Scale-up(다기관 확장)·§차별화 기술의 구매동인 논증(공단 청구 자동화·위험 예측이 must-have 동인임)에서 약속한 확장 로드맵을 실증한다. v1과 동일하게 인터넷·서버 없이 단독 구동되어야 한다.

### 과업의 범위

1) **수행 업무(In Scope)** — v1 범위 위에 **추가**되는 과업만 기술한다.
   - **법인 본부 콘솔(멀티테넌트)**: 3개 시설(행복재가복지센터·늘봄요양원·한울주야간보호센터) 테넌트 시드, 시설별 KPI 합산·비교, 행 클릭 시 해당 테넌트로 전환(`_activeTenant` 컨텍스트 스위칭).
   - **낙상 위험 예측 모델(로지스틱 회귀)**: 9개 위험요인(고령·등급·낙상이력·입원이력·다약제·인지저하·보행장애·체중감소·방문주의신호)을 가중합 → 시그모이드로 3개월 내 낙상 확률 추정. 어르신별 **상위 기여 요인(설명가능 AI)** 표시.
   - **인력배치 최적화**: LPT 휴리스틱(무거운 케이스 우선) + 비용함수(스킬 패널티·이동거리(haversine 근사)·부하)로 직원-어르신 배정 산출. 적용 전/후 부하 표준편차 개선폭 제시.
   - **수가·매출 시뮬레이션**: 정원 가동률·수가 인상률·주야간 비중 슬라이더 → 월·연 매출 실시간 투영, 보수/기본/공격 3시나리오 비교.
   - **공단 EDI 전자청구 센터(외부통합 ①)**: OAuth 2.0 Mock 인증 → EDI 전문(XML) 생성·미리보기 → 전송 → 접수번호·접수증 PDF. CSV 청구분 외부 가져오기(외부통합 ②, 파일 업로드 또는 샘플) 포함.
   - **감사 로그**: 역할·테넌트별 활동 추적, CSV 내보내기.
   - **역할 추가**: 사회복지사(social) 전용 메뉴 분기.
   - v1의 17뷰를 **23뷰**로 확장, 워크플로 유지, 신규 캡처(PC·모바일 분리) 및 검수 협조.
2) **Out of Scope(범위 외)** — v1과 동일하게 다음은 본 사이클 대상이 아니다.
   - 실 백엔드 서버·DB 구축 (멀티테넌트는 클라이언트 시뮬레이션)
   - 공단 EDI **실** 연동·실 알림톡 발송·실 OAuth/SSO (Mock 로그·Mock 토큰으로 대체)
   - 회원가입·로그인 정식 인증(역할 전환으로 우회, `CLAUDE.md` §3.4)
   - 모바일 네이티브 앱 패키징, ISMS 등 보안 인증, 개인정보 실 운영 처리·법적 자문

## 2. 과업의 세부 내용

### 2.1 시제품 사양
- **구동 형태**: 단일 HTML 파일(`v2.html`), 빌드리스·오프라인 단독 구동. `v1.html`은 보존(덮어쓰기 금지, `CLAUDE.md` §3.5). `<meta name="viewport" content="width=device-width, initial-scale=1">` 필수.
- **멀티테넌트 상태**: `_tenants`(3 시설)·`_activeTenant` 컨텍스트. 테넌트 전환 시 어르신·직원·청구·감사 로그가 해당 시설 데이터로 분기. localStorage 지속.
- **알고리즘 정확성**:
  - 낙상 위험: `p = sigmoid(intercept + Σ wᵢ·featureᵢ)`. 각 기여항 `wᵢ·featureᵢ`를 노출(설명가능). band high≥0.6 / mid≥0.35 / low.
  - 인력배치: 비용 = 스킬패널티 + 이동거리비용 + 부하비용. 최소비용 직원 배정. 개선폭 = 배정 전/후 부하 표준편차 차.
  - 수가 시뮬: 월매출 = 가동인원 × 1인 공단급여 × (1+인상률) + 본인부담. 연환산·시나리오 비교 일치.
- **EDI/OAuth Mock**: 키 없이 동작. OAuth 미인증 시 EDI 전송 버튼 비활성, 인증 후 활성. 접수번호 `LTC-YYYYMM-XXXXXX` 형식 발급.

### 2.2 사용자 UI/UX 제작 사양
- 디자인 톤: [`design-guide/디자인_애플_토스.md`](./design-guide/디자인_애플_토스.md) 준수. 모바일=[`디자인_모바일.md`], 데스크톱=[`디자인_PC.md`].
- 신규 6뷰(법인콘솔·위험예측·인력배치·수가시뮬·EDI청구·감사로그) 모두 좁은 화면(390px)에서 가로 overflow 0. EDI 전문(monospace)은 `overflow-x:auto` 코드 블록에 가둔다(페이지 overflow 금지).
- 슬라이더·표·차트가 모바일에서 스택·전폭으로 재구성. 사이드바는 모바일 바텀 탭바 + 더보기 드로어(축소판 금지). 출고 전 [`반응형_검수_필수.md`] 통과.

### 2.3 사용 부품·라이브러리·외부 의존성
- jsPDF 2.5.1 (CDN). 부재 시 mock 폴백. EDI 접수증·기존 4종 PDF 한글 깨짐 방지(캔버스 렌더).
- 차트·알고리즘은 외부 라이브러리 없이 자체 JS/SVG/CSS 구현(로지스틱 회귀·최적화 휴리스틱 포함).
- 공단 EDI·OAuth·카카오 알림톡은 Mock(키 없이 로그/토큰). API 키 하드코딩 금지(`CLAUDE.md` §3.1).

### 2.4 품질·성능 기준 (정량)
- 구현 기능 수: **120개 이상**(코드 하단 `FEATURES[]`·`window.__FEATURES.length` 검증).
- 화면/뷰: **23종 이상**(4역할 + 멀티테넌트). 다단계 워크플로 3개 이상 유지.
- **실 알고리즘 2종 이상**: 로지스틱 회귀 위험모델 + 인력배치 최적화 휴리스틱.
- **외부 시스템 통합 2건 이상**: EDI 전자청구(OAuth Mock 토큰·접수증) + CSV 외부 입출력(파일 업로드/샘플·감사로그 CSV).
- **다중 테넌트 시뮬레이션**: 3개 시설 데이터 분기·전환.
- 상태 지속성: 새로고침 후 입력·테넌트 컨텍스트 100% 유지.
- 반응형: 390px·1280px 가로 overflow 0(신규 6뷰 포함), 레이아웃 깨짐 0. 콘솔 pageerror 0.
