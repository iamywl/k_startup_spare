# 4_과업지시서 v2 — 반려동물 토탈 헬스케어 SaaS 「펫헬스북」

> 외주 용역 기업에 시제품 고도화(v2)의 일부를 위탁하기 위해 **발주자(동아리) 측이 작성**한 지시서.
> v1([`3_과업지시서_v1.md`](./3_과업지시서_v1.md)) 검수 후 추가/심화된 과업을 정의한다. **v1을 수정하지 않고 신규 파일로 분리**한다.
> 작성 근거: [`2_개발계획서.md`](./2_개발계획서.md) §3 WBS의 v2 행(9~14)을 본 문서 §1 과업의 범위로 옮겨 확정.

## 1. 과업의 개요

### 1.1 과업의 목적

1) **직접 목적** — v1의 보호자용 5개 뷰 단일 데모를, **멀티 역할(보호자·수의사·보험사) 헬스케어 플랫폼**으로 고도화한다. 본 v2 위탁의 핵심은 ① 건강 데이터 기반 **실 알고리즘 4종**(체중 추세·이상치 탐지, BCS 체형 평가·권장 사료량, 건강 위험 스코어, 견종·묘종별 접종 프로토콜 엔진), ② **3역할 RBAC 분기**(보호자/수의사/보험사 워크스페이스), ③ **외부 시스템 통합 4종 mock**(카카오 알림톡 발송 로그, 보험청구 PDF·제출·상태추적, CSV/JSON 입출력, 웨어러블 활동량 동기화), ④ 보험청구·진료기록의 다단계 워크플로 완결이다.

2) **사업 연결** — 본 과업의 산출물은 [`1_제안서.md`](./1_제안서.md)가 정의한 ⓐ 차별성(데이터 출구 확장: 건강기록 → 위험스코어 → 보험청구), ⓑ 양면 네트워크(B2B2C: 보호자 ↔ 동물병원 ↔ 보험사), ⓒ 방어가능성(데이터 해자)을 **실제 동작으로 입증**한다. 특히 역할 분기와 보험청구 상태추적은 제안서 §3.2 확장의 축 2·3과 §차별성 Moat를 구현한다.

## 2. 과업의 세부 내용

### 2.1 알고리즘 엔진(A) 사양

| 알고리즘 | 입력 | 처리 | 출력 |
|:---|:---|:---|:---|
| 체중 추세·이상치 | 체중 기록 시계열 | 이동평균(3점) + 표준편차 밴드(±2σ), 최소제곱 기울기 | 추세 라벨(증가/감소/안정), 이상치 목록(기대치 대비 이탈) |
| BCS 체형 평가 | 현재 체중, 이상 체중, 활동계수 | 체중비 구간화(저체중/이상적/과체중/비만), RER=70×kg^0.75, DER=RER×활동계수 | BCS 1~9, 권장 칼로리(kcal/일) |
| 건강 위험 스코어 | 연령·BCS·최근 증상(90일)·일정 지연 | 요인별 가중 합산 → 0~100 정규화 | 스코어, 등급(양호/주의/경계/위험), 자동 권고 문구 |
| 접종 프로토콜 엔진 | 종(견/묘), 품종 | 기본 프로토콜 + 품종 위험 가중(슬개골·척추·신장 등) 검진 항목 | 펫 등록 시 스케줄 자동 시드, 품종 맞춤 권장 검진 |

### 2.2 역할 분기(B·C·D) 사양

- **보호자**: 멀티펫 홈 대시보드 · 반려동물 · 건강기록 · 스케줄 · 건강 리포트(알고리즘) · 병원 예약 · 보험청구 · 데이터·연동 · 알림센터 (9뷰)
- **수의사**: 진료 대시보드 · 환자 차트 · 진료기록 작성 · 예약 관리 (4뷰). 진료기록 작성 시 해당 보호자 타임라인·알림에 자동 반영.
- **보험사**: 청구 대시보드(손해율) · 청구 검토(승인/반려) · 리스크 분석 (3뷰).
- 역할 전환은 사이드바(PC)·드로어(모바일)의 역할 스위처로. 역할별 포인트 컬러(보호자=인디고, 수의사=틸, 보험사=퍼플) 동적 테마.

### 2.3 외부 통합(E) 사양

- **알림톡**: D-day 도래·예약 확정·청구 결과 시 발송 페이로드 생성 → 발송 로그(타임스탬프·채널·수신자·메시지) 기록. 키 부재 시 콘솔/화면 로그 mock.
- **보험청구 상태추적**: 진료기록 선택 → 청구서 자동 구성(보장률 적용) → 제출(접수번호 발급) → 심사중 자동 전이 → 보험사 승인/반려 → 보험금 산정. 청구서 PDF 생성(jsPDF). 상태 머신: draft→submitted→review→approved/rejected.
- **CSV/JSON**: 전체 상태 JSON 백업·복원(가져오기), 건강기록 CSV 내보내기(BOM·한글 정상).
- **웨어러블**: 스마트 목걸이 활동량(걸음·활동시간) 7일치 동기화 mock → 막대 차트 시각화.

### 2.4 UI/UX 제작 사양

- 디자인 톤: Apple × Toss. 역할별 포인트 컬러 1개씩(동적 테마), WCAG AA(4.5:1) 이상.
- **반응형 필수**: `width=device-width` viewport. 모바일(≤767px) **오프캔버스 드로어**(역할 전환·전체 메뉴) + **역할별 바텀탭바**, PC(≥768px) **사이드바 256px**(역할 스위처·섹션 구분). 좁은 화면 가로 overflow 0, 차트 고정높이, 데이터 테이블은 가로 스크롤 컨테이너, 게이지는 세로 스택.
- 워크플로 3개 이상: ① 펫 등록→스케줄 자동 시드→D-day 알림톡, ② 진료기록 작성(수의사)→보호자 타임라인 반영→알림, ③ 진료 선택→청구서 PDF→제출→심사→승인/반려.
- 인증: 시제품은 **로그인 없이** 즉시 사용. 역할 전환은 데모용 스위처로(실서비스는 계정 권한 기반).

### 2.5 사용 라이브러리·의존성

- 프런트: Vanilla JS(시제품), Chart.js(CDN), jsPDF(CDN). 외부 키·서버 의존 0(오프라인 시연 보장).
- 정식 모듈: Node.js(ESM). API 키는 `.env` + `.gitignore`, 하드코딩 금지, 부재 시 mock.
- 검증 안 된 라이브러리 무분별 추가 금지, 미사용 의존성 제거.

| 상태 지속성 | 새로고침 후 펫·기록·청구·알림·로그·역할 100% 유지(localStorage) |

| 오프라인 | 외부 서버·키 없이 단일 HTML 구동(CDN 외 의존 0) |
