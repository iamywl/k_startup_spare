## 3. 환경

- 형태: 단일 `index.html` 자체완결(외부 인프라·빌드 불필요, `file://` 구동).
- 라이브러리: jsPDF 2.5.1(CDN). 그 외 표준 Web API(Canvas·localStorage·Blob/CSV). **하드코딩 시크릿 없음.**
- 캡처 환경: macOS / Playwright Chromium. PC 1280×900, 모바일 390×844(deviceScaleFactor 2, isMobile).
- 검증: 6뷰 전 화면 390px 가로 overflow 0(`body.scrollWidth == innerWidth`), 콘솔 pageerror 0.

## 4. 실행/구동 방법

1. `projects/waste-circular/index.html`를 브라우저로 연다(`file://` 또는 정적 호스팅). 로그인 불필요(시드 데이터 자동 탑재).
2. 워크플로 시연: **배출·계근**에서 배출 등록 → 운반/처리 중량 입력(체인 검증) → **전자인계서**에서 서명·발행 → **정산**에서 정산 확정·세금계산서 → **ESG 리포트** 확인·PDF → **연동**에서 올바로 매핑·CSV.
3. 캡처 재현: 워크스페이스 트리에서 `node projects/waste-circular/capture.mjs` (playwright는 워크스페이스 루트 `node_modules`로 해석).

## 5. 화면·실물 캡처

각 캡처는 *무엇을 보여주는가 / 의도 / 검토 결과* 순으로 기술한다. PC=`./captures/v1/`, 모바일=`./captures/mobile/v1/`.

### 5-1. 대시보드 (PC)
![대시보드 PC](./captures/v1/01-dashboard.png)
- 무엇: KPI 4종(입력기한 임박 2건·분기 배출량·재활용률 61.3%·미정산 처리비) + 월별 배출량 막대 + 처리방법 도넛(재활용 61%) + 처리해야 할 일 목록.
- 의도: 담당자가 첫 화면에서 "지금 무엇이 급한가(올바로 D-day)"와 ESG 지표를 한눈에 보게 한다.
- 검토: KPI·차트·할 일 목록이 시드 데이터로 정확히 집계됨. 할 일에 단계별 액션(계근/서명/올바로 D-2·D-0/정산)이 표시됨.

### 5-2. 배출·계근 (PC)
![배출·계근 PC](./captures/v1/02-discharge.png)
- 무엇: 좌측 배출 등록 폼, 우측 수거·계근(체인 중량 검증) 테이블, 하단 배출 건 목록.
- 의도: 배출 등록과 계근 입력을 한 화면에서, 중량 검증과 함께 처리.
- 검토: 폼·테이블·목록 정상 렌더, 품목·단가·처리방법이 시드와 일치.

### 5-3. 계근·체인 검증 (PC)
![계근 검증 PC](./captures/v1/03-weighing.png)
- 무엇: 배출 등록 후 운반계근(1,485)·처리계근(1,480) 입력 → 체인 오차 % 산출.
- 의도: 운반·처리 중량 불일치(정산 분쟁 원인)를 허용오차(±3%)로 자동 검증.
- 검토: 오차 0.3%대 → "정상" 칩 표시. 입력값이 localStorage에 반영됨.

### 5-4. 전자인계서 목록 (PC)
![인계서 목록 PC](./captures/v1/04-manifest-list.png)
- 무엇: 계근 완료 건의 체인 상태·인계서 발행 여부·올바로 입력기한.
- 의도: 발행 대상과 입력기한을 한 표에서 관리.
- 검토: 미발행 건에 "서명·발행" 버튼, 발행 건에 D-day·PDF 버튼.

### 5-5. 전자서명 모달 (PC)
![서명 모달 PC](./captures/v1/05-sign-modal.png)
- 무엇: 인계 요약(번호·품목·처리중량·운반사·정산 예상)과 캔버스 서명 영역.
- 의도: 수거 현장에서 폰/PC로 즉시 손서명 → 종이 인계서 대체.
- 검토: 모달이 뷰포트 내 정상 표시, 서명 영역 동작.

### 5-6. 인계서 발행 (PC)
![인계서 발행 PC](./captures/v1/06-manifest-issued.png)
- 무엇: 서명 직후 인계서 발행 + 올바로 입력기한 D-10 시작 토스트, 발행 목록의 D-day(D-0·D-2·D-6)·입력완료 칩.
- 의도: 발행 시점부터 올바로 10일 기한이 카운트다운됨을 가시화(과태료 방지).
- 검토: 새 서명 건이 발행 상태로 전환, D-day가 인계일+10일 기준으로 정확.

### 5-7. 정산 (PC)
![정산 PC](./captures/v1/07-settle.png)
- 무엇: 처리중량×단가+운반비로 산출된 정산액, 정산 대기/완료 합계, 정산 확정·세금계산서 버튼.
- 의도: 위탁계약 단가 기반 정산을 자동화하고 세금계산서까지 잇는다.
- 검토: 정산액이 단가·중량·운반비로 정확히 계산, 확정 시 상태 전환.

### 5-8. ESG 리포트 (PC)
![ESG PC](./captures/v1/08-esg.png)
- 무엇: 총 처리량·재활용률·매립률·CO₂e 추정 KPI.
- 의도: 원청·공시가 요구하는 ESG 지표를 버튼 없이 즉시 산출.
- 검토: 처리방법별 실집계 기반 재활용률·매립률·탄소가 일관되게 표시.

### 5-9. ESG 처리방법별 집계 (PC)
![ESG 상세 PC](./captures/v1/09-esg-detail.png)
- 무엇: 재활용·소각·매립별 처리량·비중·CO₂e + ESG 리포트 PDF 버튼.
- 의도: ESG 리포트의 산출 근거를 표로 투명하게 제시.
- 검토: 비중 합 100%, CO₂e가 방법별 계수로 계산됨.

### 5-10. 연동 (PC)
![연동 PC](./captures/v1/10-integration.png)
- 무엇: 올바로 매핑 내보내기(mock 로그) + CSV 입출력.
- 의도: 올바로 입력 보조·외부 데이터 연계를 시연(오프라인 mock).
- 검토: 매핑 내보내기 후 로그 기록, CSV 내보내기 동작.

### 5-11. 상태 지속성 (PC)
