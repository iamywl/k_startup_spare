## v1 한계 및 v2 개선 매핑

> v2는 v1 PoC(1천만원)의 부족함을 메워 베타/시리즈A(5억) 수준으로 끌어올리는 사이클이다. v1에서 부족했던 항목 ↔ v2 해결 방식 1:1.

| # | v1 한계 | v2 개선 | 증거 |
|---:|:---|:---|:---|
| 1 | **단일 사용자** — 한 사업장·한 사람 기준, 역할 구분 없음 | **멀티테넌트(3 사업장) + RBAC(5 역할)** — 테넌트별 데이터 분리, 역할별 표시 뷰·권한 분기 | §5-14,17 / 캡처 17-tenant-switch |
| 2 | **1인 서명** — 인계서를 한 번의 서명으로 발행 | **3자(배출·운반·처리) 순차 서명** — 각 역할만 해당 단계 서명, 3자 완료 시 발행 | §5-4,5 / 5-13 콘솔 |
| 3 | **정산 검증 없음** — 체인 오차만 표시, 정산은 무조건 가능 | **3중 정산 이상 탐지** — 체인±3% + 품목 2σ 통계 이상치 + 청구단가 불일치, **이상 건 정산 자동 보류·차단** | §5-8,9 |
| 4 | **사후 집계만** — 과거 배출량 차트뿐, 미래 예측 없음 | **배출량 예측** — 3MA + 최소제곱 추세 + 잔차 표준편차 예측구간·R², 품목별 예측 | §5-1,6 |
| 5 | **업체 고정** — 위탁사를 수동 선택, 비교 근거 없음 | **가중점수 처리업체 추천** — 단가·거리·재활용률·정산신뢰도 가중합 순위화 | §5-7 |
| 6 | **연동 2종(올바로 mock·CSV)** | **외부 통합 4종** — 올바로 신고 전송(접수번호)·계근 IoT 게이트웨이·펌뱅킹 정산 CSV 입출력·탄소 리포트 외부 제출 | §5-11 |
| 7 | **추적성 없음** — 누가 무엇을 바꿨는지 기록 없음 | **감사로그** — 배출·계근·서명·정산·연동 전 행위를 행위자·역할·테넌트·시각으로 기록 + CSV | §5-16 |
| 8 | **뷰 6종·워크플로 1개** | **뷰 12종·워크플로 3개** | 전 캡처 |

## v2 신규/심화 산출물

> v2에서 새로 추가·심화한 기능을 캡처와 함께 입증한다.

| 분류 | 신규/심화 산출물 | 핵심 |
|:---|:---|:---|
| 알고리즘 | 배출량 예측 엔진 | `forecast()` — 선형회귀 β·a, R²(SSE/SST), `next = 0.5·추세 + 0.5·3MA`, `band = √(SSE/(n-2))` |
| 알고리즘 | 정산 이상 탐지 | `verifyManifest()` — 체인 편차·`z=|x-μ|/σ ≥ 2`·`|billUnit-unit|/unit ≥ 10%` 3중 룰 |
| 알고리즘 | 처리업체 추천 | `recommend()` — `0.40·단가절감 + 0.20·근접 + 0.30·재활용률 + 0.10·신뢰도` |
| 사용자 모델 | 멀티테넌트·RBAC | 3 테넌트 데이터 분리 + 5 역할 권한 매트릭스 + `can(perm)` 권한 게이트 |
| 워크플로 | 3자 순차 전자서명 | `nextSigner()`·역할별 서명 권한·3자 완료 발행 |
| 통합 | 외부 4종 | 올바로 접수번호·IoT 디바이스 로그·펌뱅킹 CSV 왕복·탄소 접수 ID |
| 운영 | 감사로그·운반/처리 콘솔·검증보류 정산차단 | 추적성·역할 작업 분리·과청구 방지 |

## 3. 환경

- 형태: 단일 `v2.html` 자체완결(외부 인프라·빌드 불필요, `file://` 구동). v1 `index.html`과 별개 파일·별개 localStorage 키.
- 라이브러리: jsPDF 2.5.1(CDN). 그 외 표준 Web API(Canvas·localStorage·Blob/CSV·FileReader). **하드코딩 시크릿 없음.**
- 캡처 환경: macOS / Playwright Chromium. PC 1280×900, 모바일 390×844(deviceScaleFactor 2, isMobile).
- 검증: 12뷰 전 화면 390px 가로 overflow **0**(`body.scrollWidth − innerWidth`), 콘솔 **pageerror 0**.

## 4. 실행/구동 방법

1. `projects/waste-circular/v2.html`를 브라우저로 연다(`file://` 또는 정적 호스팅). 로그인 불필요(시드·역할 자동 탑재).
2. 상단 **테넌트(사업장)·역할** 셀렉트로 멀티테넌트·RBAC를 시연한다.
3. 워크플로: **배출·계근**(배출 등록→IoT 수신) → **전자인계서**(배출담당→운반기사→처리장 순차 서명) → **정산 검증**(이상 탐지) → **정산**(보류 차단·정상 정산) → **외부연동**(올바로·펌뱅킹·탄소) → **감사로그**.
4. 캡처 재현: 워크스페이스 트리에서 `node projects/waste-circular/capture-v2.mjs` (playwright는 워크스페이스 루트 `node_modules`로 해석).

## 5. 화면·실물 캡처

각 캡처는 *무엇을 보여주는가 / 의도 / 검토 결과* 순으로 기술한다. PC=`./captures/v2/`, 모바일=`./captures/mobile/v2/`.

### 5-1. 대시보드 — 예측·이상탐지 (PC)
![대시보드 PC](./captures/v2/01-dashboard.png)
- 무엇: KPI 4종(올바로 임박·분기 처리량·재활용률·**정산 이상 탐지**) + 월별 처리량 + **다음달 예측 막대(파랑·±구간 캡)** + 처리방법 도넛, 상단 테넌트·역할 셀렉트.
- 의도: 운영 현황·미래 예측·정산 리스크를 한 화면에서. v1 대비 예측·이상탐지 추가.
- 검토: KPI·차트·예측이 시드 데이터로 정확히 집계, 사이드바 3섹션(운영·인텔리전스·연동) 정상.

### 5-2. 배출·계근 + IoT 자동수신 (PC)
![배출계근 PC](./captures/v2/02-discharge-iot.png)
- 무엇: 배출 등록 폼 + 계근 IoT 자동수신 버튼 + 체인 검증 테이블 + 배출 건 목록(처리사 열).
- 의도: IoT 게이트웨이 수신으로 수기 계근 입력을 대체, 체인 재검증.
- 검토: IoT 수신 후 운반·처리 계근값이 채워지고 체인 칩이 표시됨.

### 5-3. 전자인계서 — 3자 순차 서명 (PC)
![인계서 PC](./captures/v2/03-manifest-3party.png)
- 무엇: 계근 완료 건의 체인·**3자 서명 칩(배·운·처)**·올바로 상태·다음 서명자 버튼.
- 의도: 배출-운반-처리 책임 분계를 3자 서명으로 명확화(분쟁·위변조 방지).
- 검토: 미서명 단계는 해당 역할만 서명 가능(타 역할 버튼 disabled).

### 5-4. 서명 모달 — 스텝퍼 (PC)
![서명 PC](./captures/v2/04-sign-step.png)
- 무엇: 배출자/운반자/처리자 스텝퍼(현재 단계 강조) + 인계 요약 + 캔버스 서명.
- 의도: 누가 어느 단계를 서명 중인지 가시화.
- 검토: 배출담당 역할에서 "배출자" 단계 활성, 서명 후 다음 단계로 진행.

### 5-5. 인계서 발행 (PC)
![발행 PC](./captures/v2/05-manifest-issued.png)
- 무엇: 3자 서명 완료 후 발행 상태·올바로 D-day, 인계서 PDF 버튼.
- 의도: 3자 모두 서명해야 발행됨을 입증.
- 검토: 배·운·처 칩 전부 녹색, 발행 후 올바로 D-day 카운트다운.

### 5-6. 배출량 예측 (PC)
![예측 PC](./captures/v2/06-forecast.png)
- 무엇: 다음달 예측·±예측구간·추세(kg/월)·R² KPI + 품목별 예측 막대 + 표(이동평균·추세·예측·±구간).
- 의도: 처리계약·집하 캐파 사전 확보를 위한 미래 배출량 추정.
- 검토: R²·추세·예측이 월별 실측으로 계산됨(예: R² 0.84, 추세 +1,538kg/월).

### 5-7. 처리업체 추천 (PC)
