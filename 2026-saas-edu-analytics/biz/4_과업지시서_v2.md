# 과업지시서 v2 — 클래스렌즈 학습분석 SaaS 베타 심화

> v1([`3_과업지시서_v1.md`](./3_과업지시서_v1.md)) 검수 후 추가·심화된 과업. v1을 수정하지 않고 신규 파일로 분리한다.
> 범위는 [`2_개발계획서.md`](./2_개발계획서.md) §WBS 의 v2 행에서 옮겨 확정한다.

## 1. 과업의 개요

### 1.1 과업의 목적

1) **직접 목적**: v1 PoC를 **파일럿 학교 투입·Series A 데모 가능한 베타**로 심화한다. 다중 학급/교사(테넌트), 학습분석 알고리즘(성취추세·취약개념 군집·위험학생 조기경보), AIDT/LMS 학습로그 연동 mock, 학부모 알림(알림톡 mock), 개인정보 보호(가명·동의), 역할 권한(교사·관리자·학부모), 학급 비교 차트를 구현한다.
2) **사업 연결**: [`1_제안서.md`](./1_제안서.md) §8 로드맵의 Beta(v2) 단계 산출물에 해당하며, §5.3 해자(데이터 락인)·§10 멀티테넌트 아키텍처·§10.2 개인정보 보호 설계를 실 구현으로 입증한다.

### 1.2 과업의 범위

**In Scope**

| # | 업무 항목 |
|:---:|:---|
| 1 | 다중 학급/다중 교사 멀티테넌트(학교=테넌트) 권한 분기 |
| 2 | 학습분석 알고리즘(성취 추세·취약개념 군집·위험학생 조기경보) |

| 2 | 개인정보 보호 인증 **취득**(ISMS-P 등) | v3, 발주자 주관 |

## 2. 과업의 세부 내용

### 2.1 기능 사양

| 기능 | 세부 사양 |
|:---|:---|
| 멀티테넌트 | 학교=테넌트, RLS school_id 격리. 교사는 담당 학급만, 관리자는 학교 전체 접근 |
| 학습분석 알고리즘 | (a) 성취 추세 추정(시점별 회귀/이동평균), (b) 취약개념 군집(개념 정답률 기반 클러스터링), (c) 위험학생 조기경보(점수 하락·결손 패턴 임계 탐지). 판단 근거 노출(블랙박스 금지) |
| AIDT/LMS 연동 mock | 표준 학습로그 포맷(학생·콘텐츠·시각·정오답) 파서, 클래스렌즈 학생 마스터에 매핑·인입 |
| 학부모 알림(알림톡 mock) | 리포트 생성 시 알림톡 발송 mock(키 없으면 발송 로그 기록), 폴백 SMS 경로 |
| 개인정보 보호 | 학생 식별자↔가명 분리 저장, 보호자 동의 기록·철회 관리 화면 |
| 역할 권한 | 교사·관리자·학부모 3역할 RBAC, 역할별 화면·데이터 분기 |
| 학급 비교 차트 | 반 간 성취 비교·시점 간 추세 비교 시각화 |

### 2.2 UI/UX 사양

- 화면/뷰 **8종 이상**: 교사 대시보드·취약개념 히트맵·맞춤과제·학부모 리포트(v1) + 관리자(학교 전체)·학부모(모바일)·동의 관리·학급 비교·위험학생 경보 뷰.
- 워크플로 **3개 이상**: ① 평가→분석→피드백(v1) ② 로그 인입→군집 진단→경보 ③ 동의 관리→가명처리→리포트 발송.
- 반응형: 교사 데스크톱 + 학부모 모바일 레이아웃.

### 2.3 사용 부품·라이브러리·의존성

- v1 스택 유지 + 멀티테넌트 백엔드(Node.js ESM API, RBAC), DB(PostgreSQL RLS; 데모는 fallback 허용), 군집/추세 알고리즘 구현.
- 알림톡·AIDT/LMS는 mock 인터페이스(키 부재 시 로그·시드). API 키 하드코딩 금지.

### 2.4 품질·성능 기준(정량)

| 항목 | 기준 |
|:---|:---|
| 테넌트 격리 | 타 학교 학생 데이터 조회 0건(격리 테스트 통과) |
| 알고리즘 근거 | 위험학생 경보·군집 결과에 근거 표시 100% |
| 외부 통합 mock | AIDT/LMS 로그 인입 + 알림톡 발송 로그 — 2건 실 동작(로그·CSV 입출력) |
| 역할 분기 | 교사·관리자·학부모 권한 경계 누수 0건 |
| 반응형 | 데스크톱·모바일 양 레이아웃 깨짐 0건 |
