# 디자인 톤 가이드

> Apple의 절제된 명료함 × Toss의 친근한 대담함
>
> 이 문서는 제품 전반의 시각 언어와 인터랙션 원칙을 정의합니다. 모든 화면, 컴포넌트, 문구는 이 가이드를 기준으로 판단합니다.

---

## 1. 디자인 철학

### 핵심 한 문장

**"군더더기 없이 명료하되, 차갑지 않게."**

Apple에서 가져오는 것과 Toss에서 가져오는 것을 명확히 구분합니다.

| 출처 | 가져오는 것 | 가져오지 않는 것 |
|---|---|---|
| Apple | 여백의 위계, 콘텐츠 우선(Deference), 정교한 디테일, 절제된 컬러 | 차가운 거리감, 과도한 미니멀리즘 |
| Toss | 한 화면 = 한 가지 목적, 큰 타이포, 친근한 말투, 즉각적인 피드백 모션 | 과도한 일러스트, 가벼운 농담 톤 |

### 3대 원칙

**① 명료함 (Clarity)**
모든 화면은 사용자가 0.5초 안에 "여기서 뭘 해야 하는지" 알 수 있어야 합니다. 한 화면에는 하나의 핵심 행동(Primary Action)만 존재합니다. 두 번째로 중요한 것은 과감히 다음 화면으로 미룹니다.

**② 콘텐츠 우선 (Deference)**
UI는 콘텐츠를 돋보이게 하는 무대입니다. 장식적인 테두리, 불필요한 배경색, 의미 없는 아이콘을 제거합니다. 위계는 색이 아니라 **크기·굵기·여백**으로 만듭니다.

**③ 다정한 정확함 (Warm Precision)**
픽셀 단위로 정확하되, 말투와 모션은 사람처럼 따뜻하게. 에러 메시지조차 사용자를 탓하지 않고 다음 행동을 안내합니다.

## 2. 컬러

### 원칙
- 기본은 **무채색 + 단 하나의 브랜드 블루**. 색은 "의미가 있을 때만" 사용합니다.
- 텍스트 컬러는 검정이 아니라 **약간 부드러운 잉크색**을 사용해 눈의 피로를 줄입니다.
- 상태 컬러(성공/경고/위험)는 시스템 전체에서 단 하나의 값으로 통일합니다.

### 토큰

```css
/* Brand */
--color-primary:        #3182F6;  /* Toss Blue 계열 — 핵심 액션 전용 */
--color-primary-hover:  #1B64DA;
--color-primary-light:  #E8F3FF;  /* 선택됨, 배지 배경 */

/* Neutral (Grayscale) */
--color-ink:            #191F28;  /* 본문·타이틀 텍스트 */
--color-ink-secondary:  #4E5968;  /* 보조 텍스트 */
--color-ink-tertiary:   #8B95A1;  /* 캡션, 플레이스홀더 */
--color-divider:        #E5E8EB;  /* 구분선 */
--color-surface:        #F2F4F6;  /* 카드 배경, 입력 필드 */
--color-background:     #FFFFFF;  /* 페이지 배경 */

/* Semantic */
--color-success:        #14AE5C;
--color-warning:        #FFB331;
--color-danger:         #F04452;
```

### 사용 규칙
- `--color-primary`는 **한 화면에 한 번** 등장하는 것이 이상적입니다 (주 버튼, 또는 핵심 숫자).
- 회색 5단계 이상 만들지 않습니다. 위 토큰 외의 회색이 필요하다고 느껴지면 위계 설계를 다시 검토합니다.
- 다크모드 도입 시 컬러를 단순 반전하지 않고 별도 토큰 세트를 정의합니다.

## 3. 타이포그래피

### 서체

| 역할 | 서체 | 비고 |
|---|---|---|
| 국문 | Pretendard | Toss·국내 프로덕트 표준. 자간 -1~-2% 권장 |
| 영문/숫자 | Pretendard 내 라틴 | 숫자는 `font-variant-numeric: tabular-nums`로 금액 정렬 |
| 코드/데이터 | JetBrains Mono (선택) | 개발자 대상 화면에만 |

### 타입 스케일

```css
--text-display:   font-size: 28px; font-weight: 700; line-height: 1.3;   /* 화면 타이틀 */
--text-title:     font-size: 22px; font-weight: 700; line-height: 1.35;
--text-heading:   font-size: 17px; font-weight: 600; line-height: 1.45;
--text-body:      font-size: 15px; font-weight: 400; line-height: 1.6;
--text-caption:   font-size: 13px; font-weight: 400; line-height: 1.5;  /* color: ink-tertiary */
--text-number-xl: font-size: 32px; font-weight: 800;                    /* 금액·핵심 숫자 */
```

### 규칙
- **타이틀은 크고 굵게, 망설이지 않는다.** Toss처럼 화면 상단의 질문/타이틀이 화면의 절반의 일을 합니다. 예: "얼마를 보낼까요?"
- 굵기는 400 / 600 / 700 / 800 네 단계만 사용합니다. 500은 쓰지 않습니다(위계가 흐려짐).
- 한 화면의 텍스트 크기는 **최대 3단계**까지만 섞습니다.

## 4. 레이아웃 & 여백

### 스페이싱 시스템 (4px 베이스)

```
4 / 8 / 12 / 16 / 24 / 32 / 48 / 64
```

- 컴포넌트 내부: 8~16
- 컴포넌트 사이: 16~24
- 섹션 사이: 32~48
- 화면 좌우 기본 패딩: **20px** (모바일), 콘텐츠 최대 폭 720px (데스크톱 문서형 화면)

### 규칙
- 여백이 애매하면 **더 넓게**. 좁은 여백 두 개보다 넓은 여백 하나가 위계를 만듭니다.
- 구분선보다 여백을 먼저 시도합니다. 여백으로 구분이 안 될 때만 `--color-divider`를 사용합니다.
- 카드 남용 금지. 카드는 "묶음"의 의미가 있을 때만 사용하고, 단순 나열은 리스트로 처리합니다.

## 5. 형태 — 라운드, 그림자, 표면

```css
/* Radius */
--radius-s:  8px;    /* 입력 필드, 작은 버튼 */
--radius-m:  12px;   /* 카드, 모달 내부 요소 */
--radius-l:  20px;   /* 바텀시트, 큰 카드 */
--radius-full: 999px;/* 칩, 필 버튼 */

/* Shadow — 그림자는 '떠 있는 것'에만 */
--shadow-card:  0 1px 3px rgba(0,0,0,0.04);
--shadow-float: 0 8px 24px rgba(25,31,40,0.12);  /* 바텀시트, 드롭다운, 토스트 */
```

- 평면에 붙어 있는 요소(카드, 리스트)는 그림자 대신 `--color-surface` 배경색으로 구분합니다.
- 그림자는 **레이어가 실제로 떠 있을 때만** (모달, 시트, 토스트) 사용합니다 — Apple의 깊이(Depth) 원칙.
- 테두리(border)는 입력 필드 포커스 상태 외에는 거의 사용하지 않습니다.

## 6. 모션

### 원칙
모션은 장식이 아니라 **인과관계의 설명**입니다. "내가 누른 것 → 이것이 일어났다"를 몸으로 느끼게 합니다.

```css
--motion-fast:     150ms cubic-bezier(0.3, 0, 0, 1);   /* 버튼 프레스, 토글 */
--motion-default:  250ms cubic-bezier(0.3, 0, 0, 1);   /* 화면 전환, 시트 등장 */
--motion-spring:   spring(stiffness 300, damping 30);   /* 숫자 카운트업, 강조 */
```

### 규칙
- 버튼은 누르는 순간 `scale(0.97)`로 살짝 눌리는 촉감을 줍니다 (Toss의 시그니처).
- 금액·핵심 숫자는 등장 시 카운트업 또는 스프링 모션으로 강조합니다.
- 동시에 움직이는 요소는 최대 2개. 그 이상이면 산만함입니다.
- `prefers-reduced-motion` 설정 시 모든 모션을 페이드로 대체합니다.

## 7. 컴포넌트 톤

### 버튼
- **Primary**: `--color-primary` 배경, 흰 텍스트, 높이 52px(모바일 풀폭), radius 12px, weight 600.
- **Secondary**: `--color-surface` 배경, `--color-ink` 텍스트. 테두리 없음.
- 한 화면에 Primary 버튼은 하나. 비활성 상태는 회색이 아니라 **primary의 35% 투명도**로 처리해 "곧 활성화된다"는 기대를 줍니다.

### 입력 필드
- 레이블은 필드 위 13px, 필드는 `--color-surface` 배경 + radius 12px, 포커스 시 primary 1.5px 테두리.
- 에러는 빨간 테두리 + 필드 아래 13px 안내문. 안내문은 반드시 "어떻게 고치는지"를 포함합니다.

### 바텀시트 / 모달
- 모바일의 선택·확인 동작은 모달보다 **바텀시트**를 우선합니다 (radius-l 상단, 스프링 등장).
- 시트 타이틀은 질문형으로: "어떤 계좌로 받을까요?"

## 8. 보이스 & 톤

UI 문구는 디자인의 일부입니다. Toss의 말하기 방식을 따릅니다.

| 상황 | ❌ 이렇게 쓰지 않기 | ✅ 이렇게 쓰기 |
|---|---|---|
| 화면 타이틀 | "송금 정보 입력" | "얼마를 보낼까요?" |
| 버튼 | "제출", "확인" | "보내기", "동의하고 시작하기" |
| 에러 | "오류가 발생했습니다 (E-4012)" | "잠시 연결이 끊겼어요. 다시 시도해 주세요" |
| 빈 화면 | "데이터가 없습니다" | "아직 내역이 없어요. 첫 거래를 시작해 보세요" |
| 완료 | "처리가 완료되었습니다" | "보냈어요!" |

규칙:
- **해요체**로 통일. 합쇼체("~습니다")는 약관·법적 고지에만.
- 시스템 관점이 아니라 사용자 관점의 동사를 씁니다. ("저장되었습니다" → "저장했어요")
- 사과·변명보다 다음 행동을 먼저 안내합니다.
- 느낌표는 완료·축하 순간에만, 한 화면에 한 번.

## 9. Do & Don't 체크리스트

배포 전 화면마다 확인합니다.

- [ ] 이 화면의 핵심 행동이 **하나**인가?
- [ ] 타이틀만 읽고도 무엇을 하는 화면인지 아는가?
- [ ] Primary 컬러가 한 화면에 한 번만 쓰였는가?
- [ ] 회색 텍스트가 3단계(ink / secondary / tertiary)를 넘지 않는가?
- [ ] 구분선 대신 여백으로 해결할 수 있는 곳은 없는가?
- [ ] 모든 버튼·문구가 해요체이고, 시스템 용어가 없는가?
- [ ] 눌리는 요소에 프레스 피드백(scale)이 있는가?
- [ ] 에러 문구가 "다음에 뭘 해야 하는지"를 알려주는가?

## 부록 — 적용 메모 (워크스페이스 실무)

> 본 워크스페이스의 **모든 프로젝트(`*/projects/*`)는 이 가이드를 단일 기준으로 따른다.**

### 금지 팔레트 (→ §2 토큰으로 치환)

| 금지 | 종류 | 치환 |
|---|---|---|
| `#4f46e5` · `#6366f1` | Tailwind indigo | `#3182F6` (브랜드) |
| `#1e40af` · `#2563eb` | blue-700/800 | `#1B64DA` / `#3182F6` |
| `#64748b` · `#94a3b8` | slate 텍스트 | `#4E5968` / `#8B95A1` (ink) |
| `#cbd5e1` · `#e2e8f0` | slate 라인/면 | `#E5E8EB` (divider) |
| `#0f172a` · `#1e293b` | slate 진회색 | `#191F28` (ink) |

### 적용 방식 (2026-06-10 일괄 통일)

- 6개 데모 앱 모두 Tailwind Play CDN(`cdn.tailwindcss.com`) 사용.
- 각 HTML `<head>`에 `tailwind.config` **컬러 재매핑** 주입 → `blue/indigo/sky → #3182F6`, `slate/gray → ink 그레이스케일`, `emerald/green/amber/yellow/rose/red → 시맨틱 단일값`. 기존 유틸 클래스를 무수정으로 자동 적용. 차트 카테고리 hue(cyan/violet/lime/채널색)는 의도적으로 보존.
- **Pretendard** 로드 + 첫 `<style>`에 `design-tone-components` 블록 주입: 버튼 프레스 `scale(.97)`, 비활성 `opacity:.35`, 입력 포커스 primary 1.5px 링, `button.w-full`/`a.w-full[bg]` → `min-height:52px · radius:12px`.
- **새 화면 추가 시** 위 config·컴포넌트 블록이 이미 깔려 있으므로 표준 Tailwind 클래스(`bg-blue-600`, `text-slate-600`, `w-full` 등)만 써도 톤이 자동으로 맞는다.
- 캡처 재생성: `npm i playwright@latest`(브라우저는 chromium 캐시 재사용) 후 각 앱의 `capture*.mjs`/`_cap*.mjs` 실행. fieldworker v1은 `CAPTURE_V1=1` 필요.
